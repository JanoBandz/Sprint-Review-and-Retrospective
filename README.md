# Sprint-Review-and-Retrospective

To interpret user needs, you must bridge the gap between abstract desires and technical execution by using User Stories as a functional roadmap. These stories follow a simple template which shifts the focus from purely technical features to the actual benefit provided to the end-user. By grounding development in these stories, you ensure that every line of code serves a purpose, while Acceptance Criteria provide a clear, objective definition of done. This prevents scope creep and ensures the final program aligns with what the user actually requires to solve their problem.

Approaching development requires a transition from building blocks of code to an iterative and incremental mindset. Instead of trying to perfect a complex system in one go, you break the project into a Minimum Viable Product and use Agile processes like Sprint Planning and Backlog Refinement to manage tasks. In my future work, incorporating Continuous Integration and Test-Driven Development will allow me to maintain high code quality while frequently delivering functional updates. This approach turns the overwhelming task of building a program into a series of manageable, logical "sprints" that allow for constant feedback and adjustment.

Being a strong team member within a Scrum framework relies on transparency and a team-first mentality. In daily stand-ups, success is measured not just by your individual progress, but by your willingness to communicate blockers early and support others to meet the Sprint Goal. A good developer embraces Code Reviews and Pair Programming as collaborative learning opportunities rather than personal critiques. Ultimately, being a student of Agile means moving away from solo-coding habits and toward a culture of accountability.
